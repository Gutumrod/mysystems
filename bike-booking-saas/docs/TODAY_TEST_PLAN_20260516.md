# Today Test Plan — bike-booking-saas

ไฟล์นี้คือแผนเทสสำหรับรอบถัดไปของ `bike-booking-saas`

## เป้าหมาย

- ยืนยันว่า booking แบบชั่วโมงและแบบวันทำงานได้จริงบน live
- ยืนยันว่ากฎวันหยุดของร้านบังคับถูกต้อง
- ยืนยันว่า dashboard, งานวันนี้, และการแก้ booking ยังทำงานตรงกัน
- เทสให้พอเห็นว่า flow หลักพร้อมใช้งานจริง

## กติกาก่อนเริ่มเทส

1. เปิด `START_HERE.md`
2. เปิด `SESSION_NOTES_CURRENT.md`
3. เปิดไฟล์นี้อีกครั้งก่อนเริ่มกดเทส
4. ถ้า repo หรือ live state ไม่ตรงกับ note ให้หยุดเช็กก่อน ไม่เดา
5. เทสทีละเคสและจดผลทันที

## ชุดเทส 1 — Customer booking

### 1.1 จองรายชั่วโมง

#### เคสที่ต้องลอง
- เลือกวันเปิดทำการของ KMO
- เลือก service แบบชั่วโมง
- เลือกเวลาเริ่มที่ว่างจริง
- ส่ง booking

#### ต้องเห็น
- UI ไม่ฟ้อง error
- booking ถูกสร้างจริง
- หน้า success หรือการตอบกลับจาก backend ถูกต้อง

#### สิ่งที่ต้องเช็กเพิ่ม
- ถ้าวันที่เลือกเป็นวันหยุด ร้านต้องบล็อกไม่ให้จอง
- ถ้าเวลาชน booking เดิม ระบบต้องบล็อกหรือแจ้งเต็ม

### 1.2 จองรายวัน 1 วัน

#### เคสที่ต้องลอง
- เลือก service แบบวัน
- เลือกวันเริ่มที่ร้านเปิด
- ให้วันสิ้นสุดเท่ากับวันทำการขั้นต่ำที่ระบบกำหนด
- ส่ง booking

#### ต้องเห็น
- minimum end date ถูกคำนวณตามวันทำการจริง
- ถ้าวันสิ้นสุดสั้นเกินไป ต้องโดนบล็อก
- booking ถูกสร้างจริงเมื่อวันครบ

### 1.3 จองรายวันหลายวัน

#### เคสที่ต้องลอง
- เลือกช่วงหลายวัน
- ให้ช่วงนั้นมี holiday แทรกกลาง
- ตรวจว่า UI ขยับวันสิ้นสุดออกอัตโนมัติ
- ส่ง booking

#### ต้องเห็น
- ระบบนับเฉพาะวันทำการ
- holiday ระหว่างทางทำให้วันสิ้นสุดยาวออก
- ถ้าวันจบยังสั้นเกิน ต้องแจ้งเตือนถูกต้อง

### 1.4 วันหยุดของร้าน

#### เคสที่ต้องลอง
- เลือกวันหยุดประจำร้าน
- เลือกวันหยุดเพิ่มเติม

#### ต้องเห็น
- วันหยุดแสดงชัดก่อนกดส่ง
- รายชั่วโมงต้องไม่มีเวลาว่าง
- รายวันต้องไม่อนุญาตให้เริ่มในวันหยุด

## ชุดเทส 2 — Admin / Dashboard / งานวันนี้

### 2.1 Dashboard

#### ต้องเช็ก
- คิววันนี้แสดงถูก
- คิวรายวันยาวต้องอยู่จนกว่าจะจบช่วง
- กด `ทำเสร็จ`
- กด `ยกเลิก`
- กด `No-show`

#### ต้องเห็น
- สถานะเปลี่ยนแล้วรายการหายจาก list ที่ควรหาย
- งานที่ยัง active ต้องยังอยู่

### 2.2 งานวันนี้

#### ต้องเช็ก
- booking รายวันหลายวันยังแสดงในทุกวันที่อยู่ในช่วง
- booking ที่เริ่มไปแล้วแต่ยังไม่จบ ต้องยังอยู่
- `ดู/แก้ไข` ต้องแก้ได้จริง

#### ต้องเห็น
- รายการ active ไม่หายก่อนวันจบ
- ปิดงานแล้วรายการหายตามสถานะ

### 2.3 Booking edit

#### ต้องเช็ก
- เปลี่ยนวันนัด
- เปลี่ยนเวลานัด
- เปลี่ยนวันสิ้นสุดของ daily
- บันทึกแล้วข้อมูลเปลี่ยนจริง

#### ต้องเห็น
- ฟอร์มแก้ไขไม่ใช่แค่ดูอย่างเดียว
- validation ยังกันข้อมูลผิด

## ชุดเทส 3 — Control Center

### 3.1 Shop lifecycle

#### ต้องเช็ก
- change status
- suspend
- reactivate
- delete shop

#### ต้องเห็น
- ลบร้านลง Supabase จริง
- ข้อมูลลูกโซ่หายตาม FK cascade
- ไม่มี permission denied

### 3.2 Billing / expiry

#### ต้องเช็ก
- ต่ออายุ 7 วัน
- ต่ออายุ 30 วัน
- ต่ออายุ 365 วัน
- วันครบจ่าย / วันหมดอายุ ไม่ถอยหลังผิดทาง

#### ต้องเห็น
- billing history โผล่ใน activity log
- ร้านที่ due / expired ถูกแยกเห็นได้

### 3.3 Signup / password

#### ต้องเช็ก
- สมัครร้านใหม่
- ตั้งพาสเวิร์ดเอง
- ลืมรหัสผ่าน
- รีเซ็ตรหัสผ่าน
- approve / reject ใน control

#### ต้องเห็น
- signup request เข้าคิวจริง
- approve แล้วสร้าง shop / owner mapping ได้

## ชุดเทส 4 — Verification checklist

หลังจบแต่ละเคส ให้จด:

- วันที่เทส
- เคสที่ลอง
- ผลลัพธ์
- error ที่เจอ
- root cause ถ้ารู้แล้ว
- ต้องแก้อะไรต่อ

## สิ่งที่ถ้าพบ error ใหม่ ต้องหยุดก่อน

- schema cache mismatch
- RPC signature mismatch
- permission denied
- holiday logic ตีกับ booking rules
- booking กินวันผิด

## สรุป

เทสวันนี้เน้นยืนยัน flow จริง ไม่ใช่แค่ UI ผ่าน
ถ้าผ่านครบ 3 ชุดแรก จะถือว่าระบบหลักยังนิ่งอยู่
